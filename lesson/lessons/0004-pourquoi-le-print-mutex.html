Philosophers · Leçon 04 · Data race

Pourquoi le print_mutex existe

Le deuxième dragon : l'affichage partagé. Deux raisons — dont une que les correcteurs adorent.

Ce que tu sauras faire à la fin Défendre à l'oral deux choses : (1) pourquoi printf depuis plusieurs threads est dangereux, et (2) l'astuce vraiment maligne de ton code — garder print_mutex verrouillé pendant qu'on lève stop, pour qu'aucun message ne s'affiche après died. Suite du voyage (leçon 03).

1. L'écran est une ressource partagée

Tes nb_philo threads écrivent tous sur la même sortie standard. La sortie standard, c'est comme une fourchette : une ressource commune. Sans coordination, deux threads peuvent écrire « en même temps ».

Une ligne de log ici, c'est en réalité plusieurs morceaux : timestamp, id, message, puis le retour à la ligne. Si deux threads s'entremêlent, tu peux obtenir :

42 3 is eati42 5 is thinking
ng

C'est une data race sur l'affichage : deux threads touchent une ressource commune sans verrou, et le résultat est illisible et non reproductible. (Sortie standard partagée : printf(3).)

2. La parade : sérialiser avec un mutex

Ton custom_print_timestamp (src/print.c) encadre l'affichage d'un mutex — une section critique :

void	custom_print_timestamp(t_philo *philo, const char *message)
{
	t_table	*table = philo->table;

	pthread_mutex_lock(&table->print_mutex);      // ← un seul thread entre
	if (!is_stopped(table))
		printf("%ld %d %s\n",
			get_time_ms() - table->start_time, philo->id, message);
	pthread_mutex_unlock(&table->print_mutex);    // ← le suivant peut entrer
}

Tant qu'un thread tient print_mutex, les autres attendent à la porte. Chaque ligne sort donc entière, d'un bloc, jamais coupée.

Nuance honnête (pour ne pas dire de bêtise à l'oral) Sur glibc, un printf unique est déjà protégé en interne : il ne plantera pas. Mais (a) le standard ne te garantit pas ce comportement partout, et surtout (b) ce n'est pas la seule raison du mutex ici. La vraie raison qui suit est bien plus forte.

3. La vraie raison : rien ne s'affiche après died

Le sujet est strict : une fois un philosophe mort, plus aucun message ne doit s'afficher. Pas de « is eating » en retard après le « died ». Comment ton code le garantit-il ? Regarde print_death (src/monitor.c) :

void	print_death(t_philo *philo)
{
	t_table	*table = philo->table;

	pthread_mutex_lock(&table->print_mutex);      // ← prend l'affichage
	if (!is_stopped(table))
		printf("%ld %d %s\n",
			get_time_ms() - table->start_time, philo->id, MSG_DIED);
	pthread_mutex_lock(&table->stop_mutex);
	table->stop = 1;                                // ← lève stop…
	pthread_mutex_unlock(&table->stop_mutex);
	pthread_mutex_unlock(&table->print_mutex);    // ← …AVANT de rendre l'affichage
}
Le déclic à retenir print_death écrit died et lève stop sans jamais lâcher print_mutex entre les deux. Donc aucun philosophe ne peut se glisser pour afficher un message pendant cet intervalle. Quand un philosophe obtient enfin print_mutex, il teste if (!is_stopped(table)) → c'est déjà vrai → il n'imprime rien. died est forcément la dernière ligne.

C'est une coordination à deux verrous : print_mutex protège l'écran, stop_mutex protège le drapeau, et les tenir imbriqués crée une garantie que ni l'un ni l'autre ne donnerait seul.

5. À toi de jouer

Deux expériences dans src/print.c :

  • Expérience 1 — commente les lock/unlock de custom_print_timestamp, recompile, lance ./philo 5 800 200 200 avec beaucoup de philosophes. Observe les lignes qui se mélangent. Puis remets le mutex.
  • Expérience 2 (mentale) — imagine qu'on relâche print_mutex avant de lever stop dans print_death. Quel message pourrait alors se faufiler après died ? Sache le raconter — c'est une question de correcteur classique.
Source primaire à lire

man pthread_mutex_lock(3) — la sémantique exacte du verrou : un seul détenteur à la fois, les autres bloquent jusqu'au unlock. C'est précisément ce qui sérialise ton affichage. Relis aussi le chapitre « Mutual Exclusion » du Little Book of Semaphores.
