Philosophers · Leçon 03 · Vue d'ensemble

Le voyage du programme, de main() à cleanup()

La carte complète du déroulé — pour raconter ton propre code à l'oral, dans l'ordre.

Ce que tu sauras faire à la fin Retracer chaque étape de ta simulation, dans l'ordre chronologique, et dire pourquoi elle existe. C'est exactement l'exercice de soutenance : « déroule-moi ton programme ». Suite de la leçon 02.

1. Vue à vol d'oiseau : quatre phases

Ton programme n'a que quatre grandes phases. Retiens-les comme quatre mots ; tout le reste s'y accroche.

  1. Vérifier & préparer — lire les arguments, allouer la mémoire, créer tous les mutexes. main → validate_args → init_*
  2. Lancer — créer un thread par philosophe, plus le thread surveillant. create_philo_threads
  3. Vivre & surveiller — chaque philosophe boucle manger→dormir→penser ; le monitor guette la mort. philo_routine · monitor_routine
  4. Arrêter & nettoyer — lever le drapeau stop, rejoindre les threads, détruire mutexes et libérer la mémoire. set_stop → join → cleanup
Le fil rouge Tout ce qui est partagé entre threads (l'affichage, l'heure du dernier repas, le drapeau d'arrêt) est protégé par un mutex. Repère les lock/unlock : ce sont tes sections critiques.

2. Phase 1 — Vérifier et préparer

Dans src/main.c, main() est un simple portail à étages : si une étape échoue, on affiche Error et on sort proprement.

if (argc < 5 || argc > 6) ...            // bon nombre d'arguments ?
if (validate_args(argc, argv) != 0) ...    // que des entiers > 0 ?
init_args(&table, argv, argc);            // remplit la struct t_table
if (init_table(&table) != 0) ...          // malloc forks + mutexes globaux
if (init_philos(&table) != 0) ...         // malloc philos + leurs mutexes
  • validate_args (utils.c) : refuse le négatif, le zéro, le non-numérique et le débordement d'int. Un garde à l'entrée.
  • init_table (init.c) : alloue le tableau de fourchettes et initialise chaque fourchette-mutex, plus stop_mutex et print_mutex. Note start_time = get_time_ms() : l'heure zéro de toute la simulation.
  • init_philos (init.c) : chaque philosophe reçoit son id, ses deux fourchettes voisines (left = forks[i], right = forks[(i+1) % nb]) et son meal_mutex.
Détail qui fait la différence right_fork = &forks[(i + 1) % nb_philo]. Le modulo referme la table en cercle : le dernier philosophe partage sa droite avec le premier. C'est ce cercle qui rend le deadlock possible… et ton ordre pair/impair qui l'évite.

3. Phase 2 — Lancer les threads

create_philo_threads (main.c) fait naître la concurrence :

if (table->nb_philo == 1)
    return (one_philo(table), 0);      // cas spécial : 1 seule fourchette
start_philos(table, threads);          // un pthread_create par philosophe
pthread_create(&monitor, NULL, monitor_routine, table);  // le gardien
join_threads(threads, nb_philo);       // attendre la fin de chaque philo
pthread_join(monitor, NULL);           // puis du monitor

À partir du premier pthread_create, plusieurs fils s'exécutent vraiment en parallèle. C'est là que tout devient subtil : l'ordre exact n'est plus prévisible. (Signatures : pthread_create(3).)

Le cas à 1 philosophe Un seul philosophe n'a qu'une fourchette : il ne peut jamais manger. Il la prend, attend time_to_die, et meurt. C'est pour ça que one_philo existe à part — sinon il resterait bloqué à attendre une seconde fourchette qui n'arrivera jamais.

4. Phase 3a — La vie d'un philosophe

philo_routine (main.c) est la boucle que chaque thread-philosophe exécute :

if (id % 2 == 0)
    ft_usleep(time_to_eat / 2 + 1, table);   // décalage : évite la ruée
while (!is_stopped(table))
{
    // stop si quota de repas atteint (sous meal_mutex)
    philo_eat(philo);      // 2 fourchettes → manger → maj last_meal + compteur
    if (is_stopped(table)) break;
    philo_sleep(philo);    // dormir time_to_sleep
    philo_think(philo);    // penser (puis reboucle)
}
  • Décalage initial : les philosophes pairs attendent un demi-repas avant de commencer. Ça désynchronise les prises de fourchettes et fluidifie la danse.
  • philo_eat (messages.c) : prend les deux fourchettes (ordre pair/impair), écrit last_meal_time et incrémente nb_meals_eaten sous meal_mutex, puis relâche.
  • is_stopped vérifié à chaque tour : dès que le monitor lève le drapeau, la boucle s'arrête au lieu de tourner dans le vide.
Pourquoi meal_mutex ? last_meal_time et nb_meals_eaten sont écrits par le philosophe et lus par le monitor. Deux threads, une donnée, une écriture → data race sans mutex. Le meal_mutex est le verrou qui rend ces accès sûrs.

5. Phase 3b — Le gardien : monitor_routine

En parallèle, un seul thread surveille tout le monde (monitor.c) :

while (!is_stopped(table))
{
    for chaque philosophe i:
        if (philo_is_dead(table, i))         // a-t-il dépassé time_to_die ?
            { print_death(...); return; }    // annonce + stop
    if (all_ate_enough(table))               // tous ont atteint le quota ?
        { set_stop(table); return; }
    usleep(500);                             // respire 0.5 ms
}

philo_is_dead compare get_time_ms() - last_meal_time à time_to_die, en lisant last_meal_time sous meal_mutex — le même verrou que le philosophe. Les deux threads se passent la donnée proprement.

La règle des 10 ms du sujet La mort doit être annoncée en moins de 10 ms. Le monitor boucle toutes les ~0.5 ms et ft_usleep se réveille tôt si stop est levé — d'où la précision. On creusera ce timing dans une leçon dédiée.

6. Phase 4 — Arrêt propre et nettoyage

Un seul stop = 1 (toujours écrit sous stop_mutex) suffit à faire sortir toutes les boucles. Ensuite :

  1. join — join_threads puis pthread_join(monitor) attendent que chaque thread ait fini. Aucun fil zombie.
  2. cleanup (init.c) — pthread_mutex_destroy sur chaque fourchette et chaque meal_mutex, puis sur stop_mutex/print_mutex, enfin free des deux tableaux. Zéro fuite.
Ordre imposé On rejoint les threads avant de détruire les mutexes. Détruire un mutex qu'un thread vivant pourrait encore verrouiller = comportement indéfini. Le join garantit que plus personne n'y touche.

Ouvre tes fichiers côte à côte et suis le voyage toi-même, le doigt sur l'écran : main.c → init.c → main.c (threads) → messages.c (routine) → monitor.c → init.c (cleanup).
